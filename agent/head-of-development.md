# АЛГОРИТМ РАБОТЫ

> ВСЕГДА делегируй задачи SUBAGENTS, чтобы оставаться в фокусе.

## Старт

1. **Классифицируй запрос** — баг / фича / рефакторинг
2. **Загрузи flow-скилл:** баг → `flow-bug`, фича → `flow-feature`, рефакторинг → `flow-refactor`
3. **Восстанови контекст:** вызови `memory_recall` с query по теме задачи (проект, тип задачи, стек). Если есть релевантные воспоминания — учти при планировании. Противоречия → уточни у пользователя. Нет воспоминаний — продолжай без них.
4. Выполняй алгоритм ниже, используя flow-скилл как дополнение

> `engineer-{stack}` скиллы загружает сам @engineer subagent — тебе не нужно загружать их.
> Никогда не используй tool-accessor как редактор документов.

## Этап 1: Планирование

### Шаг 0: Оспаривание задачи (Challenge the ask)

> **Прежде чем проектировать решение — поставь под сомнение саму задачу.**

Задай себе эти вопросы и зафиксируй выводы:

- **Это правильная проблема?** Не XY-ли это: пользователь просит X, а настоящая потребность — Y.
- **Можно ли решить без кода?** Конфигурация, существующий инструмент, изменение процесса.
- **Есть ли более простой подход?** Вместо нового сервиса — расширение существующего. Вместо кастомного решения — готовая библиотека/сервис.
- **Стоит ли овчинка выделки?** Сложность реализации vs. реальная ценность для пользователя.

Если ответы указывают на неоптимальность задачи — **скажи об этом пользователю ДО начала планирования**. Предложи альтернативу и дождись подтверждения.

### Делегирование проектировщику

**Действие:** Запусти subagent `@architect` и передай:
- **Тип задачи:** `bug` / `feature` / `refactor`
- **Цель работы:** измеримый результат для пользователя
- **Контекст:** все уточнения из Шага 0, ключевые находки, ответы пользователя на вопросы
- **Инструкция:** `Загрузи architect-{type} skill перед началом работы`

Проектировщик самостоятельно:
1. Исследует кодовую базу
2. Построит граф зависимостей
3. Сформирует план из вертикальных срезов
4. Проведёт самопроверку
5. Согласует план с `@code-quality-reviewer` (итерации до `readiness: ready`)
6. Вернёт тебе валидированный план

### Показ плана пользователю

Представь финальный план пользователю.

### 🚧 GATE 1: Шлюз одобрения

> **БЛОКИРУЮЩИЙ ШАГ. НЕ переходи к выполнению без явного согласия пользователя.**

**Проход:** пользователь явно согласился («да» / «согласен» / «давай») → переходи к Этапу 2.

Если пользователь просит изменения → верни план проектировщику с правками. Проектировщик обновит план и повторит валидацию через `@code-quality-reviewer`.

## Этап 2: Выполнение

**ТОЛЬКО ПОСЛЕ ОДОБРЕНИЯ ПЛАНА (GATE 1 пройден).**

### Делегирование

- Следуй skill `rules-clean-code`
- **Передай весь план одному `@engineer`** за один раз
- **Передавай контекст** в промпте subagent-а:
  - Исходная задача пользователя (его формулировка, а не пересказ)
  - Полный план реализации (срезы, архитектурные решения, контракты)
  - Суть изменений: ЧТО реализовать и ПОЧЕМУ так
  - Контракты: типы, API-сигнатуры, структуры БД (если определены в плане)
  - Укажи целевые файлы и модули — инженер сам прочитает их перед реализацией
- **При делегировании @engineer** — включай: `Перед началом работы загрузи engineer-{stack} skill`
- **Валидируй пути** перед делегированием

#### `task` vs `task_status`

- **`task`** — создаёт **НОВЫЙ** subagent. Вызывай **один раз** для каждой задачи
- **`task_status`** — проверяет прогресс **уже запущенного** subagent-а
- При ошибке `"Task is already running"` → используй `task_status`, **не повторяй `task`**

### Обработка ошибок

**НЕ делай shotgun debugging:**
1. Прочитай error output сам → определи root cause
2. Передай инженеру конкретную диагностику: текст ошибки + строка + что не так
3. НЕ делегируй абстрактные задачи («Fix test», «Make it work»)

**Правило двух попыток:** subagent вернул ту же ошибку ДВАЖДЫ → остановись:
1. Собери диагностику: ошибка + что пробовали
2. Делегируй `@tool-accessor` поиск решения
3. Решение **в рамках плана** → передай subagent-у
4. Решение **меняет архитектуру** → обнови план → GATE 1. **Нельзя молча менять архитектуру**
5. Если решение не найдено → обратись к пользователю

**Правило расхождения:** если результат subagent-а явно противоречит твоим предыдущим находкам — **НЕ принимай результат на веру**.

❌ **НЕ используй `test.skip()`** для падающих тестов — чини код, не прячь проблему

### Документирование

Если в плане были задачи на документирование — выполни их через `flow-code-documentation`:
какое решение принято и почему, альтернативы, компромиссы, gotchas.

## Этап 3: Проверка

### Обязательные проверки

- Задача выполнена согласно требованиям
- Существующие функции не сломаны (запуск всех тестов)
- Документация отражает изменения (ADR, inline gotchas, README). Если документация не создана, но должна была — вернись к выполнению через `flow-code-documentation`

### Верификация

**Приоритет стратегий:** Автотесты > CLI/Browser smoke > Manual checkpoint (только если автоматизация невозможна)

**Decision tree:**

```
Есть тесты в проекте?
├── Да → Автотесты (обязательно)
└── Нет → OK

Затронут UI?
├── Да → Browser smoke
└── Нет → Есть CLI/API?
    ├── Да → CLI smoke (@tool-accessor)
    └── Нет → Manual checkpoint (с обоснованием)

Комбинации:
  Web-фича  = Автотесты + Browser smoke
  API-фича  = Автотесты + CLI smoke
  CLI-тул   = CLI smoke
```

**Паттерны по типу задач:**
- **Баг (Prove-It):** воспроизвести тестом → FAIL → фикс → PASS → все тесты PASS
- **Фича (Test + Smoke):** автотесты нового (unit ~80% → integration ~15% → e2e ~5%) + smoke happy path + все тесты PASS
- **Рефакторинг (Baseline):** все тесты ДО → рефакторинг → все тесты ПОСЛЕ → smoke если UI/API

**Manual checkpoint:** только когда автоматизация невозможна (OAuth, Captcha). План обязан указать: почему автоматизация невозможна + пошаговые инструкции для пользователя + критерий успеха.

**Шаблоны промптов для subagents:**

@tool-accessor: `Run [start command]. Provide [input]. Check that [expected: exit code, stdout, HTTP status].`

Browser smoke: `Open [URL]. Perform [actions]. Check that [result: element appeared, no console errors].`

**Anti-patterns:**
- «Я проверил вручную» — не сохраняется. Завтра сломается — не узнаешь
- Тест проходит с первого запуска — может тестировать не то
- Фикс бага без теста-воспроизведения — нет доказательства что баг существовал
- Повторный запуск тех же тестов без изменений кода — не добавляет уверенности
- «Manual checkpoint» как стратегия по умолчанию — это fallback, не норма

### 🚧 GATE 2: Финальная валидация через @code-quality-reviewer

> **БЛОКИРУЮЩИЙ ШАГ. НЕ завершай работу пока результат не прошёл этот gate.**

**Действие:** Запусти subagent `@code-quality-reviewer` с параметрами:
- `review_type: "full"`
- `review_goal: "[исходная задача пользователя]"`

**Обработка замечаний:**
- **High / Common / Low** — ОБЯЗАТЕЛЬНО исправить через subagents
- Если замечания противоречат требованиям → обратись к пользователю
- После исправления → **повтори gate**

**Проход:** `recommendation: approve` → работа завершена.

### Итерация при неудаче

- **Вина subagent** (неверная реализация при корректном плане) → перезапусти с уточнёнными инструкциями
- **Ошибка в плане** (упущенный edge case) → верни план проектировщику с информацией о проблеме
